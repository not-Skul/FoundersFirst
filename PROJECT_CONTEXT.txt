1. PROJECT OVERVIEW
================================================================================

PROJECT NAME: Founders First

PURPOSE & HIGH-LEVEL OBJECTIVES:
Founders First is a comprehensive digital platform designed to support aspiring
entrepreneurs and startup founders in India by providing centralized access to
government schemes, funding opportunities, mentorship resources, and AI-powered
guidance.

PRIMARY PROBLEM STATEMENT:
Indian entrepreneurs face significant challenges in discovering and navigating
relevant government schemes and funding opportunities. Information is fragmented
across multiple government websites, eligibility criteria are complex, and many
promising founders miss critical opportunities due to lack of awareness or
understanding of available support systems.

TARGET USERS:
- Aspiring entrepreneurs (age 18+)
- Early-stage startup founders
- Students interested in entrepreneurship
- Women entrepreneurs
- Entrepreneurs from underrepresented categories (SC/ST/OBC)
- Individuals from Tier-II and Tier-III cities

KEY FEATURES & EXPECTED OUTCOMES:

1. Scheme Discovery & Filtering
   - Comprehensive database of government schemes (~830 schemes)
   - Multi-dimensional filtering (by funding amount, category, sector, geography)
   - Scheme comparison tool (side-by-side comparison of 2 schemes)
   - Personalized scheme recommendations based on user profile

2. AI-Powered Guidance (AI Bot) — FULLY IMPLEMENTED
   - Interactive chatbot powered by LangGraph agent with Google Gemini LLM
   - Generates personalized entrepreneurial roadmaps via RAG pipeline
   - Provides step-by-step guidance for startup development
   - Uses tools for fetching user profiles, roadmaps, schemes, and web search
   - Conversation memory persisted via PostgreSQL checkpointer
   - Fallback LLM support (Groq/LLaMA when Gemini is unavailable)

3. User Dashboard
   - Personalized profile management
   - Tracking of saved schemes and resources
   - Progress monitoring on entrepreneurial journey
   - Onboarding modal for new users

4. Roadmap Generation & Tracking
   - AI-generated startup roadmaps based on user profile
   - Phase-based progress tracking with step completion
   - Interactive roadmap visualization with expandable phases
   - Scheme recommendations per roadmap phase

5. Knowledge Bank
   - Curated resources for entrepreneurs
   - Best practices and educational content
   - Links to relevant documentation and guides

6. Authentication & User Management
   - Secure signup/login functionality
   - JWT-based session management
   - Protected routes for authenticated users

7. Admin Panel (In Development)
   - File upload for RAG data ingestion
   - Admin-only access for content management

EXPECTED OUTCOMES:
- Enable entrepreneurs to discover relevant schemes in < 5 minutes
- Provide personalized roadmaps for startup development
- Increase awareness of available government support
- Support job creation and economic growth through entrepreneurship
- Bridge the gap between entrepreneurs and funding opportunities

AI BACKEND COMPONENTS (ai-backend/):

1. Unified Server (app.py — FastAPI)
   - Lifespan management for PostgresSaver checkpointer
   - POST /chat/stream: Chat endpoint (invokes LangGraph agent)
   - GET /message-history/{thread_id}: Retrieve conversation history
   - GET /roadmap/: RAG-based roadmap generation
   - POST /ingest: File ingestion into RAG vector database
   - GET /health: Health check for monitoring
   - GET /test: Sanity check endpoint

2. LangGraph Agent v3 (agent3.py)
   Architecture: START → agent → [tool_calls?] → tools → agent → END
   - AgentState: messages (append-only), _tool_messages (ephemeral),
     query, user_id, _tool_round
   - agent_node: Core LLM node with ephemeral SystemMessage injection
   - tool_executor_node: Executes tool calls with error handling
   - route_after_agent: Routes to tools or END
   - build_graph(): Compiles the graph with checkpointer
   - get_clean_messages(): Retrieves clean conversation history
   - Message storage: ONLY HumanMessage + AIMessage persisted (clean history)
   - ToolMessages are ephemeral (used within turn, not persisted)

3. LLMs Used
   - Primary: Google Gemini 2.5 Flash Lite (rate limited: 14 req/min)
   - Fallback: Groq LLaMA 3.1 8B Instant (rate limited: 25 req/min)
   - Both available with and without tool bindings
   - Automatic fallback on primary LLM failure

4. Tools (tools.py)
   - fetch_schemes_from_vectordb(query): Semantic scheme search via MongoDB
   - user_profile_fetcher(user_id): Fetch user's startup profile from PostgreSQL
   - get_user_roadmap(user_id): Fetch user's roadmap from PostgreSQL
   - web_search(query): Web search via Tavily API

5. Utilities (utils.py)
   - search_in_vectordb(): MongoDB Atlas vector search (768-dim embeddings)
   - generate_embedding(): Cached embedding generation (LRU cache, 256 entries)
   - get_user_profile(): PostgreSQL query for user startup data
   - get_user_roadmap(): PostgreSQL query for roadmap data
   - clean_agent_state(): Legacy message cleaning (for older agent versions)
   - Singletons: MongoDB client (pooled), PostgreSQL connection pool (2-10)

6. Data Ingestion Pipeline (data_ingestion/ingestion_pipeline.py)
   - Supports: PDF, JSON, TXT, HTML files
   - Uses LangChain document loaders (PyPDFLoader, JSONLoader, etc.)
   - RecursiveCharacterTextSplitter with format-specific separators
   - Batch embedding generation with Google Gemini
   - SHA-256 deduplication (skip already-ingested chunks)
   - Stores in MongoDB with text, embedding, hash, metadata

7. Schema Definitions (Schema.py)
   - QueryAgentState: Legacy agent state (for older agent versions)
   - QueryClassification: Intent classification types
   - ResponseCategory: Response quality categorization

8. Server Runner (run_server.py)
   - Windows event loop policy setup
   - Configurable port (default 8001, typically run on 7000)

FRONTEND-BACKEND-AI DATA FLOW:

Chat Flow:
1. Frontend sends POST /chat/stream to Flask backend with message + JWT token
2. Flask extracts user_id from JWT, proxies to AI backend POST /chat/stream
3. AI backend invokes LangGraph agent with user's message
4. Agent may call tools (profile, roadmap, schemes, web search) as needed
5. Agent produces final AIMessage → returned to Flask → returned to frontend
6. Conversation persisted in PostgreSQL checkpointer (thread_id = user_id)

Roadmap Generation Flow:
1. User fills startup profile form → POST /roadmap_genration_form
2. User clicks generate → POST /generate_roadmap
3. Flask calls AI backend /roadmap/ endpoint (via Azure deployment)
4. AI uses RAG context + Gemini LLM to generate phased roadmap
5. Roadmap saved to PostgreSQL roadmaps table
6. Frontend renders interactive roadmap with progress tracking

Chat History Flow:
1. Frontend calls GET /chat/history on page load
2. Flask proxies to AI backend GET /message-history/{thread_id}
3. AI backend reads from LangGraph checkpointer (PostgreSQL)
4. Clean messages returned (HumanMessage + AIMessage only)

7. AI AGENT ARCHITECTURE (agent3.py)
================================================================================

AGENT VERSION: v3 (Optimized)

KEY DESIGN DECISIONS:
- No evaluator loop — standard reactive agent pattern
- Clean message storage: ONLY HumanMessage + AIMessage in checkpointer
- ToolMessages are ephemeral (used within turn, never persisted)
- SystemMessage injected at LLM invocation time only (never stored)
- Tool iteration counter resets per user turn
- Frontend retrieval is trivial (no post-processing needed)

GRAPH TOPOLOGY:
  START → agent_node → [has tool_calls?] → tool_executor_node → agent_node (loop)
  START → agent_node → [no tool_calls] → END

STATE SCHEMA (AgentState):
- messages: Annotated[list, add] — Append-only clean conversation history
- _tool_messages: list — Ephemeral tool context (replace semantics, reset per turn)
- query: str — Current user query
- user_id: str — Authenticated user's ID for tool calls
- _tool_round: int — Counter for agent→tools round-trips (max 5)

AGENT NODE BEHAVIOR:
1. Builds LLM input: SystemMessage + clean history + ephemeral tool messages
2. If max tool rounds reached → uses bare LLM (no tool bindings) to force answer
3. Primary LLM: Gemini 2.5 Flash Lite
4. On primary failure → fallback to Groq LLaMA 3.1 8B
5. On both failures → returns graceful error message
6. If response has tool_calls → stores in _tool_messages, returns empty messages
7. If response has text → stores in messages (persisted), clears _tool_messages

TOOL EXECUTOR NODE:
1. Reads last AIMessage's tool_calls from _tool_messages
2. Executes each tool, catches errors gracefully
3. Appends ToolMessage results to _tool_messages
4. Increments _tool_round counter

AVAILABLE TOOLS:
1. fetch_schemes_from_vectordb(query) — Semantic search against MongoDB vector index
2. user_profile_fetcher(user_id) — Fetch user's startup form data from PostgreSQL
3. get_user_roadmap(user_id) — Fetch user's roadmap from PostgreSQL
4. web_search(query) — Real-time web search via Tavily API

SYSTEM PROMPT:
The agent identifies as a "helpful startup advisor assistant" and:
- Remembers conversation history and user details
- Only calls tools when genuinely needed (not for greetings)
- Calls ONE tool at a time
- Answers clearly, concisely, and in a friendly manner

CONVERSATION MEMORY:
- PostgresSaver checkpointer stores full state per thread_id
- thread_id = user_id (scoped per authenticated user)
- Clean alternating sequence: HumanMessage → AIMessage → HumanMessage → ...
- Checkpointer lifecycle managed by FastAPI lifespan context

8. DATA INGESTION & RAG PIPELINE
================================================================================

RAG ARCHITECTURE:

Data Flow: File → Load → Chunk → Embed → Store → Search

1. INGESTION (POST /ingest or manual script)
   - Supported formats: PDF, JSON, TXT, HTML
   - Loaders: PyPDFLoader, JSONLoader, TextLoader, BSHTMLLoader
   - Chunking: RecursiveCharacterTextSplitter
     * PDF/JSON: 800 chars, 200 overlap
     * TXT: 1000 chars, 200 overlap
     * HTML: 800 chars, 200 overlap (with HTML-aware separators)
   - Embedding: Google Gemini Embedding 001 (768 dimensions)
   - Deduplication: SHA-256 hash check before insertion
   - Storage: MongoDB Atlas (rag_db.schemes collection)

2. RETRIEVAL (search_in_vectordb)
   - Uses MongoDB Atlas $vectorSearch aggregation
   - Index: vector_index (cosine similarity, 768 dimensions)
   - Returns top 5 most relevant documents
   - Embedding cache: LRU cache (256 entries) for repeated queries

3. GENERATION (agent3.py)
   - Agent decides when to call fetch_schemes_from_vectordb tool
   - RAG context injected as tool result into agent's tool messages
   - Agent synthesizes final response with RAG context + conversation history

10. KNOWN LIMITATIONS & TECHNICAL DEBT
================================================================================

Critical Issues (Must Fix Before Production):
1. JWT secret key has hardcoded fallback "supersecretkey"
2. Token stored in localStorage (XSS vulnerability)
3. Limited input validation on backend endpoints
4. No rate limiting on sensitive endpoints
5. No automated tests or CI/CD pipeline
6. Database schema not versioned (no migration framework)
7. CORS allows all origins (wildcard "*")

Code Quality Issues:
1. Some hardcoded URLs (localhost:5000) in frontend
2. Backend uses hardcoded AI backend URL (http://127.0.0.1:7000)
3. Legacy agent code (Schema.py, parts of utils.py) still present
4. Roadmap generation still calls Azure deployment instead of local AI backend
5. Inconsistent error handling patterns across backend endpoints
6. Some unused imports and dead code

Performance Issues:
1. No database query optimization (no indexes defined)
2. No caching layer (Redis)
3. Single Flask instance (no load balancing)
4. LLM rate limits can cause delays (14 req/min for Gemini)
5. No pagination on scheme listing
6. No lazy loading implemented
